# יחסי גומלין מפורטים בין טוקני המערכת

## מבוא

מסמך זה מפרט את היחסים הלוגיים-ערכיים בין 15 טוקני הליבה של המערכת. כל יחס מוגדר לפי:
- **כיוון הזרימה** (→ חד-כיווני, ↔ דו-כיווני, ← תלות הפוכה)
- **עוצמת ההשפעה** (1-10)
- **תנאי הפעלה** (תמיד / מותנה / בעת חירום)
- **אילוץ אתי** (האם יש override מוסרי)

---

## חלק א': יחסי הגנה-בסיסית (Protection Core)

### קבוצה: T01 ↔ T06 ↔ T13 ↔ T15

זוהי הליבה ההגנתית של המערכת - "המשולש המוגן".

#### T01 (Data) → T06 (Security)
- **כיוון:** Data זקוק ל-Security
- **עוצמה:** 9/10
- **תנאי:** תמיד
- **אילוץ:** אין Data ללא Security
- **הסבר:** כל נתון שנוצר או נשמר חייב להיות מוגן מיד

#### T06 (Security) ← T01 (Data)
- **כיוון:** Security מגן על Data
- **עוצמה:** 9/10
- **תנאי:** תמיד
- **משמעות:** הגנה על שלמות ופרטיות הנתונים

#### T01 (Data) → T13 (Rights)
- **כיוון:** Data משפיע על Rights
- **עוצמה:** 8/10
- **תנאי:** תמיד
- **אילוץ:** שימוש ב-Data לא יכול להפר Rights
- **הסבר:** נתונים אישיים קשורים לזכויות יסוד (פרטיות, כבוד)

#### T13 (Rights) ← T01 (Data)
- **כיוון:** Rights שולט על שימוש ב-Data
- **עוצמה:** 8/10
- **כלל:** Data על אנשים דורש הסכמה

#### T15 (Seriousness) → T01 (Data)
- **כיוון:** מצב Seriousness מגביר הגנה על Data
- **עוצמה:** 10/10
- **תנאי:** כאשר T15 פעיל
- **השפעה:** הגבלת גישה, הקפאת שינויים, ביקורת מוגברת

#### T06 (Security) → T13 (Rights)
- **כיוון:** Security מגן על Rights
- **עוצמה:** 9/10
- **תנאי:** תמיד
- **דוגמה:** הצפנה מגנה על פרטיות, authentication מגן על זהות

#### T13 (Rights) ← T06 (Security)
- **כיוון:** Rights קובע מדיניות Security
- **עוצמה:** 9/10
- **דוגמה:** זכות הפרטיות קובעת רמת הצפנה נדרשת

#### T15 (Seriousness) → T06 (Security)
- **כיוון:** Seriousness מחזק Security
- **עוצמה:** 10/10
- **תנאי:** במצב חירום
- **השפעה:** מעבר ל"מצב נעול" עם אבטחה מוגברת

#### T15 (Seriousness) → T13 (Rights)
- **כיוון:** Seriousness מופעל להגנה על Rights
- **עוצמה:** 10/10
- **תנאי:** זיהוי הפרה או סיכון
- **מנגנון:** הפרת Rights → הפעלה אוטומטית של T15

## חלק ב': יחסי תשתית-רשתית (Infrastructure Relations)

### קבוצה: T02 ↔ T03 ↔ T04 ↔ T14

#### T02 (Network) → T03 (Compute)
- **כיוון:** Network מאפשר Compute מבוזר
- **עוצמה:** 7/10
- **תנאי:** כאשר יש פעילות רשת
- **הסבר:** ללא רשת, חישוב מוגבל למכונה מקומית

#### T02 (Network) → T04 (Storage)
- **כיוון:** Network מאפשר אחסון מבוזר
- **עוצמה:** 7/10
- **תנאי:** כאשר יש פעילות רשת
- **הסבר:** גיבויים, שכפול, ארכיון מרוחק

#### T03 (Compute) ↔ T04 (Storage)
- **כיוון:** דו-כיווני - סימביוזה
- **עוצמה:** 8/10
- **תנאי:** תמיד
- **הסבר:**
  - Compute צורך נתונים מ-Storage
  - Compute שומר תוצאות ב-Storage
  - ללא אחד מהם, השני לא יעיל

#### T03 (Compute) → T07 (Automation)
- **כיוון:** Compute מאפשר Automation
- **עוצמה:** 7/10
- **תנאי:** כאשר Compute זמין
- **אילוץ:** Automation לא יכול לפעול ללא משאבי חישוב

#### T04 (Storage) → T12 (Learn)
- **כיוון:** Storage מאפשר Learning
- **עוצמה:** 8/10
- **תנאי:** כאשר יש נתונים ב-Storage
- **הסבר:** "ללא זיכרון אין למידה" - T12 צורך מ-T04

#### T14 (Commons) → T02 (Network)
- **כיוון:** Commons מספק רשת משותפת
- **עוצמה:** 5/10
- **תנאי:** תמיד
- **הסבר:** תשתית הרשת היא משאב ציבורי

#### T14 (Commons) → T03 (Compute)
- **כיוון:** Commons מספק משאבי חישוב משותפים
- **עוצמה:** 5/10
- **דוגמה:** cloud computing ציבורי, grid computing

#### T14 (Commons) → T04 (Storage)
- **כיוון:** Commons מספק אחסון משותף
- **עוצמה:** 5/10
- **דוגמה:** ארכיונים ציבוריים, ספריות דיגיטליות

## חלק ג': מעגל הבקרה המוסרית (Moral Control Loop)

### המשולש: T10 ↔ T11 ↔ T12

זהו הלב הפועם של הבקרה האתית במערכת.

#### T10 (Measure) → T11 (Monitor)
- **כיוון:** Measure מגדיר מה Monitor עוקב אחריו
- **עוצמה:** 8/10
- **תנאי:** תמיד
- **הסבר:** "מה שלא נמדד, לא מנוטר"

#### T11 (Monitor) → T12 (Learn)
- **כיוון:** Monitor מספק נתונים ל-Learn
- **עוצמה:** 8/10
- **תנאי:** תמיד
- **הסבר:** הניטור הוא הכניסה ללמידה

#### T12 (Learn) → T08 (Govern)
- **כיוון:** Learn מעדכן את Govern
- **עוצמה:** 7/10
- **תנאי:** לאחר מחזור למידה
- **הסבר:** "מתרגמת טעויות לפרוטוקולים מחודשים"

#### T08 (Govern) → T10 (Measure)
- **כיוון:** Govern קובע מדיניות מדידה
- **עוצמה:** 6/10
- **תנאי:** לאחר עדכון Govern
- **הסבר:** "הבחירה במדד קובעת מה נחשב וראוי"

**המעגל המלא:**
```
Monitor (T11) מזהה מצב
  ↓
Measure (T10) מכמת ומעריך
  ↓
Learn (T12) מפיק לקחים
  ↓
Govern (T08) מעדכן מדיניות
  ↓
Standardize (T09) מיישם תקנים חדשים
  ↓
Automation (T07) מתעדכן
  ↓
חזרה ל-Monitor
```

#### יחסי תמיכה נוספים:

**T10 (Measure) ← T11 (Monitor)**
- ניטור מספק את הנתונים למדידה

**T10 (Measure) ← T04 (Storage)**
- מדידה צורכת נתונים היסטוריים

**T11 (Monitor) ← T05 (Identity)**
- ניטור דורש זיהוי מי עשה מה

**T11 (Monitor) ← T07 (Automation)**
- אוטומציה חייבת ניטור (אכיפה!)

## חלק ד': יחסי ממשל-זהות (Governance Relations)

### קבוצה: T05 ↔ T08 ↔ T09

#### T05 (Identity) → T08 (Govern)
- **כיוון:** Identity נדרש לכל החלטת Govern
- **עוצמה:** 6/10
- **תנאי:** לכל החלטת ממשל
- **אילוץ:** **חובה** - אין ממשל אנונימי
- **הסבר:** "מי" מחליט חשוב כמו "מה" מוחלט

#### T05 (Identity) → T11 (Monitor)
- **כיוון:** Identity נדרש לכל פעולת Monitor
- **עוצמה:** 6/10
- **תנאי:** לכל פעולת ניטור
- **אילוץ:** **חובה** - אין ניטור אנונימי
- **הסבר:** עקיבות - "מי ביצע פעולה ומהלך"

#### T08 (Govern) → T09 (Standardize)
- **כיוון:** Govern קובע תקנים
- **עוצמה:** 7/10
- **תנאי:** לאחר החלטת Govern
- **הסבר:** מדיניות → יישום בתקנים

#### T09 (Standardize) → T07 (Automation)
- **כיוון:** Standards מנחים Automation
- **עוצמה:** 6/10
- **תנאי:** לאחר קביעת תקן
- **הסבר:** אוטומציה צריכה כללים ברורים

#### T09 (Standardize) → T10 (Measure)
- **כיוון:** Standards מגדירים מה נמדד וכיצד
- **עוצמה:** 7/10
- **תנאי:** תמיד
- **אילוץ אתי:** "אם התקינה מוטה ולא הוגנת, גם המדידה תיצור עוול"

## חלק ה': יחסי הפעלה וביקורת (Execution & Oversight)

### T07 (Automation) ← T15, T10, T11

זוהי קבוצת האילוצים על האוטומציה.

#### T15 (Seriousness) → T07 (Automation) [שלילי]
- **כיוון:** Seriousness **מגביל** Automation
- **עוצמה:** 10/10 (מקדם: 0.5×)
- **תנאי:** כאשר T15 פעיל
- **השפעה:** **השהיה מיידית** של אוטומציה אגרסיבית
- **הסבר:** "במצב בקרה, דיוק ואחריות גוברים על מהירות"

#### T10 (Measure) → T07 (Automation)
- **כיוון:** Measure בוחן Automation
- **עוצמה:** 6/10
- **תנאי:** תמיד
- **הסבר:** מודדים ביצועים, שגיאות, השפעה של אוטומציה

#### T11 (Monitor) → T07 (Automation)
- **כיוון:** Monitor עוקב אחר Automation
- **עוצמה:** 6/10
- **תנאי:** **חובה**
- **אילוץ:** "אין אוטומציה ללא ניטור"
- **מנגנון:** אם Monitor מזהה חריגה → הפעלת T15 → עצירת T07

#### T07 (Automation) → T11 (Monitor)
- **כיוון:** Automation יוצר אירועים ל-Monitor
- **עוצמה:** 6/10
- **הסבר:** כל פעולה אוטומטית מתועדת ומנוטרת

כאשר טוקנים מסוימים פועלים **ביחד**, השפעתם מוכפלת.

### סינרגיות חיוביות

|-------|--------|------|--------------|-------|
| משולש מוסרי | T10, T11, T12 | ×2.5 | 1:1:1 | חייבים לפעול כמכלול |
| משולש טכני | T02, T03, T04 | ×2.0 | 1:2:2 | תלוי ב-Network |

| צמד בסיס | T01, T15 | ×2.0 | 1:1 | Data מוגן ב-Seriousness |

## חלק ט': זרימת המידע - מפת Data Flow

### זרימה ראשית (Primary Flow)

```
אירוע חיצוני / משתמש
    ↓
T01 (Data) - נתונים נקלטים
    ↓
T06 (Security) - בדיקת אבטחה
    ↓
T05 (Identity) - זיהוי מקור
    ↓
T04 (Storage) - שמירה ללוג
    ↓
T11 (Monitor) - ניטור האירוע
    ↓
T10 (Measure) - מדידת השפעה
    ↓
T12 (Learn) - הפקת לקחים (אם רלוונטי)
    ↓
T08 (Govern) - עדכון מדיניות (אם נדרש)
    ↓
T09 (Standardize) - עדכון תקנים
    ↓
T07 (Automation) - עדכון אוטומציה
```

### זרימת חירום (Emergency Flow)

```
אירוע חריג
    ↓
T11 (Monitor) מזהה
    ↓
T10 (Measure) מעריך חומרה
    ↓
    ↓ (אם חמור)
    ↓
T15 (Seriousness) מופעל
    ↓
    ├→ T07 (Automation) - עצירה
    ├→ T03 (Compute) - הגבלה
    ├→ T06 (Security) - חיזוק
    └→ T13 (Rights) - הגנה מוגברת
    ↓
T12 (Learn) חוקר שורש
    ↓
T08 (Govern) מתקן
    ↓
T15 משוחרר (רק לאחר תיקון)
```

## סיכום: עקרונות היחסים

### 1. עקרון התלות המוחלטת
- T13 (Rights) ← תמיד גובר
- T15 (Seriousness) ← תמיד שולט
- T06 (Security) ← תמיד נדרש

### 2. עקרון המעגליות
- T10 ↔ T11 ↔ T12 ← מעגל סגור של בקרה
- T03 ↔ T04 ← סימביוזה טכנית

### 3. עקרון ההיררכיה
- שכבות עליונות **תמיד** שולטות בתחתונות
- אין "חריגים למצב חירום" שמאפשרים לעקוף Rights

### 4. עקרון השקיפות
- כל זרימה מתועדת ב-T04 (Storage)
- כל זרימה מנוטרת ב-T11 (Monitor)
- כל זרימה מזוהה ב-T05 (Identity)

### 5. עקרון הכפל והגיבוי
- T03 ↔ T04 מגבים זה את זה
- T10 ↔ T11 מאמתים זה את זה
- T06 + T15 מספקים הגנה כפולה
