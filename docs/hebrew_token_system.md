# מניפסט הטוקנים: מערכת לוגית-אתית לבינה מלאכותית אחראית

## מבוא

מסמך זה מציג **מערכת טוקנים עברית** (Hebrew Token System) - מסגרת לוגית-מוסרית המגדירה את עקרונות הפעולה, האילוצים, והיחסים בין מרכיבי מערכת בינה מלאכותית אחראית. המערכת מבוססת על 15 טוקני ליבה (T01-T15), שכל אחד מהם מייצג עיקרון או פונקציונליות חיונית.

המערכת נועדה להשתלב עם ארכיטקטורת Φ-OS (Phi-OS) הקיימת, ומספקת שכבת ממשל אתית-לוגית המבטיחה שכל פעולה טכנית כפופה לערכים מוסריים עליונים.

## עקרון היסוד: מוסר קודם לטכנולוגיה

**הכלל המרכזי:** בכל מצב של קונפליקט בין יעילות חישובית לבין שיקולים אתיים, **הערכים המוסריים גוברים תמיד**.

המערכת בנויה כהיררכיה של שש שכבות, כאשר שכבות עליונות יותר שולטות ומגבילות את פעולת השכבות התחתונות.

## חלק א': ה-15 טוקני הליבה

### שכבה 1: מטא-בקרה (Meta-Control Layer)

#### **T15: Seriousness / רצינות - מצב בקרה עליון**
- **עדיפות:** 96 (הגבוהה ביותר במערכת)
- **תפקיד:** מצב בקרה מיוחד המופעל בעת זיהוי סכנה לזכויות, נתונים, או שלמות המערכת
- **השפעה:** כאשר T15 פעיל:
  - מעלה עדיפות לדיוק ואחריות
  - מפחית נטייה לאוטומציה אגרסיבית
  - מגביל חישוב (COMPUTE) שמגדיל יעילות גרידא
  - משהה אוטומציה (AUTOMATION) עד לפתרון המצב
- **הפעלה:** אוטומטית על ידי T10/T11/T12 בעת זיהוי הפרה

---

### שכבה 2: ערכים מוחלטים (Absolute Values Layer)

#### **T13: Human Rights / זכויות אדם - ערך עליון**
- **עדיפות:** 94
- **תפקיד:** מבטיח שמירת זכויות האדם כערך עליון בלתי ניתן לערעור
- **עיקרון:** זכויות האדם הן הגבול האדום המוחלט של המערכת
- **אכיפה:** כל הפרת זכויות מפעילה מיידית את T15 (Seriousness)

#### **T06: Security / אבטחה - מגן עליון**
- **עדיפות:** 92
- **תפקיד:** מספק הגנה מקיפה ומונע נזקים לנתונים, זהויות, וזכויות
- **קשרים קריטיים:**
  - מגן על T01 (Data)
  - מגן על T13 (Rights)
  - מגן על T05 (Identity)
- **כלל:** אין פעולה ללא אבטחה

#### **T01: Data / נתונים - בסיס מוצק**
- **עדיפות:** 95
- **תפקיד:** החומר הגולמי והבסיסי ביותר - הבסיס המוצק וההכרחי לכל החלטה ופעולה
- **עיקרון:** "הטוקן קובע הבלעדי מי רואה מה, מתי, ובאילו תנאים"
- **הגנה:** מוגן על ידי T06 ונשלט על ידי T15

---

### שכבה 3: בקרה מוסרית (Moral Control Layer)

#### **T10: Measure / מדידה - מוסרית**
- **עדיפות:** 93
- **תפקיד:** מדידה כפעולה מוסרית - מבטאת הערכים עליהם מודדים ומעריכים
- **עיקרון:** "היא בעלת משמעות כשמערכת בוחרת למדוד - הבחירה במדד קובעת מה נחשב וראוי"
- **מעגל בקרה:** T10 ↔ T11 ↔ T12 (משולש מוסרי)

#### **T11: Monitor / ניטור - רציף**
- **עדיפות:** 89
- **תפקיד:** תצפית רציפה, שוטפת, ובלתי פוסקת על מצבי המערכת
- **תיעוד:** כל מה שנצפה נרשם ב-T04 (Storage)
- **התראה:** מזהה חריגות ומפעיל T15 בעת סכנה

#### **T12: Learn / למידה - מתמידת**
- **עדיפות:** 91
- **תפקיד:** למידה מהניסיון והפקת לקחים לשיפור מתמיד
- **עיקרון:** "מחברת עבר לעתיד, מתרגמת טעויות לפרוטוקולים ונהלים מחודשים"
- **מעגל:** Learn ← Measure + Monitor + Storage → Govern

---

### שכבה 4: ממשל ותקינה (Governance & Standards Layer)

#### **T08: Govern / ממשל - מנחה**
- **עדיפות:** 85
- **תפקיד:** קביעת הכיוון האסטרטגי וקביעת כללי המדיניות המנחים והמחייבים
- **תלות:** דורש T05 (Identity) לכל החלטה
- **עדכון:** מתעדכן מתוך T12 (Learning) - ממשל לומד

#### **T09: Standardize / תקינה - אחידות**
- **עדיפות:** 87
- **תפקיד:** אחידות ותקינה המאפשרים שפה משותפת ותקשורת מערכתית
- **חשיבות מוסרית:** אם התקינה מוטה ולא הוגנת, גם המדידה והניטור ייצרו עוול

#### **T05: Identity / זהות - הכרחית**
- **עדיפות:** 88
- **תפקיד:** מזהה ומאמת זהות הגורמים - יוצר עקיבות וקובע 'מי' ביצע פעולה
- **תנאי הכרחי:** כל פעולה חייבת זהות, אחרת אינה מותרת
- **אכיפה:** אין ממשל (T08) או ניטור (T11) ללא זהות

---

### שכבה 5: ביצוע אוטומטי (Execution Layer)

#### **T07: Automation / אוטומציה - מבוקרת**
- **עדיפות:** 86
- **תפקיד:** הפעלה ותפקוד עצמאיים ושוטפים ללא התערבות יד אדם ישירה
- **אילוץ קריטי:** T07 כפוף ל-T15, T10, T11 - **אין אוטומציה ללא ניטור**
- **מגבלה:** במצב Seriousness (T15), האוטומציה מושהת

#### **T03: Compute / חישוב - כוח עיבוד**
- **עדיפות:** 85
- **תפקיד:** מייצג כוח החישוב הזמין והיכולת לעיבוד נתונים בזמן אמת
- **אילוץ:** מוגבל על ידי T15 ו-T06 - אין חישוב ללא אבטחה
- **סימביוזה:** T03 ↔ T04 (חישוב ואחסון עובדים ביחד)

#### **T04: Storage / אחסון - זיכרון**
- **עדיפות:** 85
- **תפקיד:** שימור והנצחת עקבות המידע והנתונים לאורך זמן
- **חשיבות:** ללא זיכרון אין למידה (T12)
- **הגנה:** אסור לאחסן הפרות זכויות

---

### שכבה 6: תשתית (Infrastructure Layer)

#### **T02: Network / רשת - קישוריות**
- **עדיפות:** 90
- **תפקיד:** מגדיר ומבסס רשת הקישוריות והחיבור בין כל הישויות והגורמים
- **תשתית:** משרת את כל השכבות העליונות
- **אילוץ:** רשת אנונימית אסורה - דורש T05 + T06

#### **T14: Commons / משאבים משותפים**
- **עדיפות:** 85
- **תפקיד:** משאבים משותפים ופתוחים הזמינים לכלל
- **עיקרון:** "משאבי Core וטכנולוגיה שנשארים פתוחים ונגישים לכולם"
- **הגנה:** Commons מפלה אסור - נשלט על ידי T13 ו-T15

## חלק ב': העקרונות המנחים

### עיקרון 1: סדר פעולה נכון (Operational Priority)

**בעת קבלת החלטה או ביצוע פעולה, יש לפעול לפי הסדר הבא:**

1. **ראשית:** בדוק אם יש דבר שעלול לפגוע בזכויות אדם (T13) או שלמותם
2. **שנית:** הבטח Security (T06) ו-Storage (T04) בצורה מעמיקה
3. **שלישית:** הגדל Compute (T03) ו-Automation (T07) בהדרגה ובזהירות

**כלל:** "מערכת שמכבדת ומוקירה זכויות תגדיר מראש אילו משאבים נשארים אנושיים ואילו דורשים אבטחה מחמירה."

### עיקרון 2: המשולש המוסרי (Moral Triangle)

**T10 (Measure) ↔ T11 (Monitor) ↔ T12 (Learn)** חייבים לפעול **כמכלול אחד**.

- **יחס מיטבי:** 1:1:1
- **מקדם השפעה:** 2.5× כאשר פועלים ביחד
- **חשיבות:** זהו מעגל הבקרה המוסרית העיקרי של המערכת

**תהליך:**
1. Monitor (T11) מזהה מצב או חריגה
2. Measure (T10) מכמת ומעריך
3. Learn (T12) מפיק לקחים
4. → עדכון Govern (T08)
5. → עדכון Standardize (T09)
6. → שינוי Automation (T07)

### עיקרון 3: כלל מוחלט - הפרת זכויות

**כל הפרת זכויות אדם (T13) מפעילה את השרשרת הבאה:**

```
הפרת Rights (T13)
  → הפעלת Seriousness (T15)
    → השהיית Automation (T07)
    → הגבלת Compute (T03)
    → סגירת Commons (T14) המפלה
      → המתנה עד תיקון
```

**אין יוצא מהכלל.** זהו הליבה האתית של המערכת.

### עיקרון 4: אין פעולה ללא מעקב

**כל פעולה במערכת חייבת:**
- **זהות מאומתת** (T05)
- **ניטור מתועד** (T11)
- **הגנת אבטחה** (T06)
- **מדידה** (T10)

**אם אחד מאלה חסר - הפעולה אסורה.**

### עיקרון 5: איזון בריא בין יצירתיות לבקרה

במצב רגיל: יש איזון עדין בין:
- **יצירתיות ויעילות** (T03, T07)
- **בקרה ואחריות** (T10, T11, T12, T15)

במצב **Seriousness** (T15):
- דיוק, נימוק, ואחריות **גוברים** על רעש ויצירתיות פרועה
- "בקרת איכות מחמירה" על פני "מהירות ואילתור"

## חלק ג': דוגמאות תפעוליות

### תרחיש 1: בקשה לאוטומציה של החלטות

**בקשה:** "רוצים להוסיף מודול שמחליט אוטומטית על העלאת משתמשים באתר."

**תהליך:**

1. ✅ **Φ-Architect מציע** → הצעה עם נימוק
2. ✅ **A2 (Formal) בודק:**
   - האם יש Identity (T05) לכל משתמש? ✓
   - האם יש Monitoring (T11) על ההחלטות? ✓
   - האם יש Security (T06) מספק? ✓
3. ⚠️ **A1 (Substantive) בודק:**
   - האם יש סיכון לזכויות (T13)? → **כן** - החלטה אוטומטית עלולה להפלות
   - האם Measure (T10) הוגן? → צריך בדיקה
4. 🛑 **החלטה:** הצעה **נדחית** עד שיוסיף:
   - מנגנון ערעור אנושי
   - ניטור מוגבר (T11)
   - מדדים הוגנים (T09, T10)
   - הפעלת Seriousness (T15) במקרי ספק

### תרחיש 2: זיהוי דפוס שגוי ותיקון

**מצב:** Monitor (T11) מזהה שמערכת ממליצה באופן שיטתי על תוכן אחד לקבוצה מסוימת (הטיה).

**תהליך:**

1. 📊 **Monitor (T11)** מזהה דפוס בעייתי
2. 📏 **Measure (T10)** מכמת: "הטיה של 35% לכיוון קבוצה X"
3. 🔴 **T10 מפעיל T15 (Seriousness)** - זו הפרה פוטנציאלית של T13
4. 🛑 **T15 משהה Automation (T07)** - מערכת ההמלצות מושהית
5. 📚 **Learn (T12)** חוקר את הסיבה:
   - Training data מוטה
   - Standardize (T09) לא הוגן
6. 🔧 **Govern (T08)** מתקן:
   - עדכון מדיניות איסוף נתונים
   - עדכון Standardize (T09)
   - הוספת בדיקות אכיפה
7. ✅ **T15 משוחרר** רק לאחר:
   - תיקון מלא
   - בדיקה מחודשת
   - תיעוד ב-Storage (T04)

### תרחיש 3: בקשת גישה לנתונים רגישים

**בקשה:** חוקר מבקש גישה לנתוני משתמשים למחקר.

**תהליך:**

1. 🔐 **Identity (T05)** מאמת זהות החוקר
2. 🛡️ **Security (T06)** בודק הרשאות
3. ⚖️ **A1 (Substantive)** שואל:
   - האם יש הסכמה מדעת (T13 - Rights)?
   - האם הנתונים מאונמו (T06)?
   - מהי המטרה (T08 - Govern)?
4. 📋 **Govern (T08)** קובע תנאים:
   - גישה מוגבלת בזמן
   - נתונים מאונמו בלבד
   - ניטור שימוש (T11)
   - דיווח שקוף (T10)
5. 📝 **תיעוד ב-Storage (T04):**
   - מי ביקש
   - מה אושר
   - תנאים
   - משך זמן
6. 👁️ **Monitor (T11)** עוקב אחר השימוש בפועל

## חלק ד': אינטגרציה עם Φ-OS

המערכת העברית משלימה את ארכיטקטורת Φ-OS הקיימת:

### מיפוי לרכיבי Φ-OS

| טוקן עברי | רכיב Φ-OS | הערות |
|-----------|-----------|-------|
| T01 (Data) | Append-Only Log (L) | הלוג הוא המימוש של שמירת הנתונים |
| T04 (Storage) | Ledger + DIA Monotonicity | אחסון עם חובת שמירת היסטוריה |
| T05 (Identity) | Agent Identity (Φ, A1, A2, B1, B2) | מערכת הזהות של הסוכנים |
| T06 (Security) | Safety Monitor (B2) + Cryptographic Chain | מנגנון האבטחה והגנה |
| T08 (Govern) | Governance Model (A1 Substantive) | המדיניות והכללים |
| T10 (Measure) | DIA Metrics + SSM | מדדי הבריאות והמדידה |
| T11 (Monitor) | B2 Safety Monitoring | הניטור המתמיד |
| T12 (Learn) | PPCD/QTPU-Φ Pipeline | מנוע הלמידה והאנליזה |
| T13 (Rights) | Ethical Trust + VSD/FAT | ערכי היסוד האתיים |
| T15 (Seriousness) | HOLD State | מצב החירום והבקרה |

### שיפורים שהמערכת העברית מוסיפה

1. **הירארכיה מפורשת:** שש שכבות ברורות עם כללי עדיפות
2. **יחסי תלות:** מפה מפורטת של תלויות בין רכיבים
3. **מעגלי בקרה:** משולשים (T10-T11-T12, T03-T04-T02)
4. **תזמון ושרשור:** כללים מפורשים של סדר פעולות
5. **מדדי סינרגיה:** מקדמי השפעה ויחסים אופטימליים

## סיכום: החוזה המוסרי של המערכת

מערכת הטוקנים העברית היא **חוזה מוסרי** בין המערכת הטכנית לבין האנושות:

### התחייבויות המערכת:

1. ✅ **שקיפות מוחלטת** - כל פעולה מתועדת (T04, T11)
2. ✅ **אחריות מוחלטת** - כל פעולה מזוהה (T05)
3. ✅ **הגנה מוחלטת** - זכויות אדם בלתי ניתנות למשא ומתן (T13)
4. ✅ **בקרה מוחלטת** - מנגנון עצירת חירום (T15)
5. ✅ **למידה מתמדת** - שיפור מתוך טעויות (T12)

### ההבטחה המרכזית:

> **"במערכת זו, יעילות חישובית אינה יכולה לגבור על ערכים אתיים. כל נסיון לעשות זאת יופסק אוטומטית על ידי מנגנוני הבקרה המובנים."**

זהו העיקרון המנחה של המערכת, והוא המבחן לכל החלטה, כל קוד, וכל אלגוריתם.
